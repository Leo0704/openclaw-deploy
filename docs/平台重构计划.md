# 跨平台结构拆分计划

## 1. 背景

当前 `apps/cli/src` 采用的是“一套 Node/TypeScript 代码跨平台运行”的组织方式。  
这种方式在项目早期有利于快速交付，但随着 Windows、macOS、Linux 的差异问题不断增加，平台相关逻辑已经分散到多个业务文件里，导致：

- Windows 问题容易混在通用逻辑里，定位慢。
- 离线包、平台专项打包、自更新策略难以独立演进。
- 路径、命令、解压、依赖安装、更新行为的边界不清晰。
- 平台专项测试很难落地。

本计划的目标不是拆成三套产品，而是保留一套核心逻辑，同时把平台差异集中管理。

## 2. 目标

- 保持一套核心业务逻辑，不复制三份代码。
- 把 `macOS / Windows / Linux` 差异集中到平台层。
- 把“运行时逻辑”和“打包/发布逻辑”分开。
- 为后续 Windows 离线包、平台专项测试、自更新策略调整提供清晰边界。
- 迁移过程尽量渐进式，避免一次性大重构。

## 3. 设计原则

- 核心业务不直接判断 `os.platform()`。
- 平台差异统一通过平台适配层暴露。
- 打包、发布、离线包属于 `packaging`，不混入核心部署逻辑。
- 运行态与 OpenClaw 交互属于 `runtime`，不直接承载平台实现细节。
- 先抽象边界，再迁移实现，第一阶段以“行为不变”为准。

## 5. 模块职责划分

### 5.1 `core/`

只放平台无关的业务流程和编排逻辑，例如：

- API 路由
- 配置处理
- 部署流程编排
- Web UI 状态和渲染
- 启动流程
- 授权和业务校验

### 5.2 `runtime/`

只放与 OpenClaw 运行态交互的逻辑，例如：

- gateway 启停
- OpenClaw 配置读写
- skills / channels 状态读取
- 运行态命令调用封装

### 5.3 `platform/`

只放系统差异：

- 默认安装路径
- 路径归一化
- Windows 长路径支持检测
- 非法字符规则
- 浏览器打开命令
- Git / pnpm / corepack / PowerShell / tar / unzip 的调用偏好
- 环境变量读取与平台默认值
- 密钥与凭证存储位置
- 管理员权限检测与权限提升判断
- PowerShell 执行策略检测
- 日志、缓存、状态、临时文件目录
- 系统代理读取与网络诊断
- 防火墙提示
- 自启动与后台守护能力
- 信号处理与子进程清理策略
- 文件锁检测与延迟删除策略
- ANSI 与终端编码能力探测
- 崩溃报告和平台诊断信息采集
- 自更新目标位置
- 解压工具
- 平台专项健康检查

### 5.4 `packaging/`

只放发布和离线包逻辑：

- release 镜像源
- 资产命名
- 不同平台的产物布局
- 离线 bundle 结构
- 非代码资源组织方式
- 图标、模板、附带配置文件
- 自更新资产下载规则

### 5.5 `shared/`

纯通用工具：

- 网络请求
- 错误模型
- 进程执行基础能力
- 资源清单与资产定位的通用辅助

## 6. 平台适配层接口建议

建议在 `apps/cli/src/platform/types.ts` 定义统一接口：

```ts
export interface PlatformAdapter {
  id: 'windows' | 'macos' | 'linux';
  getDefaultInstallPath(): string;
  normalizeProjectPath(projectPath: string): string;
  validateInstallPath(projectPath: string): { valid: boolean; error?: string };
  getBrowserOpenCommand(url: string): { file: string; args: string[] } | null;
  getDependencyInstallPlan(name: 'git' | 'pnpm'): { command: string; manual: string } | null;
  getArchiveExtractPlan(format: 'zip' | 'tar.gz'): { file: string; args: string[] } | null;
  getManagedSelfInstallTarget(execPath: string): {
    installRoot: string;
    targetExecPath: string;
    metadataPath: string;
  };
  hasAdminPrivileges(): boolean;
  getExecutionPolicyStatus(): {
    supported: boolean;
    status?: string;
  };
  isLongPathSupportEnabled(): {
    supported: boolean;
    enabled?: boolean;
  };
  getStoragePaths(appName: string): {
    configDir: string;
    cacheDir: string;
    dataDir: string;
    logDir: string;
    tempDir: string;
  };
  getSecretStore(): {
    kind: 'keychain' | 'credential-manager' | 'file';
    basePath?: string;
  };
  getProxySettings(): {
    envKeys: string[];
    systemSource?: string;
  };
  getAutostartStrategy(): {
    kind: 'launchd' | 'task-scheduler' | 'systemd' | 'none';
  };
  sendSignal(pid: number, signal: 'SIGINT' | 'SIGTERM' | 'SIGKILL'): {
    success: boolean;
    strategy: string;
  };
  cleanupChildProcesses(parentPid?: number): {
    success: boolean;
    strategy: string;
  };
  isFileLocked(filePath: string): {
    supported: boolean;
    locked?: boolean;
  };
  scheduleDeleteOnReboot?(filePath: string): {
    success: boolean;
  };
  isAnsiSupported(): boolean;
  getTerminalEncoding(): string | null;
  collectCrashContext(): Record<string, unknown>;
  getReleaseAssetNames(): {
    binary: string;
    offlineBundle?: string;
  };
}
```

第一版不用一次性实现所有方法，但接口方向应该先定下来。

### 6.1 资源与配置文件组织

建议显式区分这几类非代码资产：

- `packaging/assets/`
  - 图标、App 元数据、安装包模板、说明书模板。
- `packaging/config/`
  - 发布时附带的默认配置模板。
- `core/web-ui/`
  - Web UI 相关模板和样式。
- `shared/assets/`
  - 资源路径解析、资产清单索引、运行时查找逻辑。

建议约束：

- 平台特有资源不直接散落在 `assets/` 根目录。
- 所有要进入 release 的非代码文件都通过资产清单显式列出。
- 后续若要增加 Windows 离线包图标、默认配置、说明文档，应从 `packaging/assets/` 组装。

### 6.2 存储、日志、缓存与临时目录

这些路径不应继续散落在业务代码里，建议归属到 `platform/storage/` 和 `platform/temp/`：

- 配置目录
- 日志目录
- 数据目录
- 缓存目录
- 临时目录
- 崩溃转储目录

典型平台映射示意：

- macOS
  - `~/Library/Application Support`
  - `~/Library/Logs`
  - `~/Library/Caches`
- Windows
  - `%APPDATA%`
  - `%LOCALAPPDATA%`
  - `%TEMP%`
- Linux
  - `~/.config`
  - `~/.local/share`
  - `~/.cache`
  - `/tmp`

### 6.3 环境变量与密钥管理

需要明确分层：

- `core/`
  - 只消费抽象后的配置与凭证接口。
- `platform/security/`
  - 负责读取和写入平台特定凭证存储。

建议分阶段：

- 第一阶段
  - 继续文件存储，但路径由平台层统一提供。
- 第二阶段
  - Windows 预留 Credential Manager 适配器。
  - macOS 预留 Keychain 适配器。
  - Linux 仍允许文件存储作为默认实现。

### 6.4 网络、代理与防火墙

建议在 `platform/network/` 明确下面能力：

- 读取系统代理或环境变量代理
- 输出用户可理解的代理诊断信息
- Windows 防火墙拦截提示
- 平台网络错误信息标准化

### 6.5 自启动、后台守护与崩溃采集

虽然当前还没有完整开机自启需求，但目录和接口建议现在就预留：

- `platform/autostart/`
  - macOS `launchd`
  - Windows `Task Scheduler`
  - Linux `systemd --user`
- `platform/diagnostics/`
  - 平台版本信息
  - 进程树和 PATH 信息
  - 崩溃前上下文采集

### 6.6 Windows 权限、执行策略与长路径

建议把这组问题明确归入 `platform/security/` 与 `platform/process/`：

- 管理员权限检测
  - 某些目录写入、注册表变更、系统级设置修改可能需要提升权限。
- PowerShell ExecutionPolicy 检查
  - 用于判断脚本执行是否可能被系统策略阻止。
- 长路径支持检测
  - Windows 长路径支持未开启时，深层目录或 `node_modules` 结构可能触发路径上限问题。

这组能力建议至少预留下面接口：

- `hasAdminPrivileges()`
- `getExecutionPolicyStatus()`
- `isLongPathSupportEnabled()`

### 6.7 信号处理、文件锁与进程清理

这组问题建议归入 `platform/process/` 与 `platform/update/`：

- 跨平台信号语义差异
  - Windows 不等价支持 Unix 风格信号。
- 子进程清理差异
  - Windows 常需要 `taskkill` 或树状清理策略。
- 文件锁差异
  - Windows 上文件被占用时不能直接删除、重命名或覆盖。
- 自更新替换差异
  - 运行中的可执行文件无法直接替换，可能需要延迟删除或重启后清理。

建议预留接口：

- `sendSignal(pid, signal)`
- `cleanupChildProcesses()`
- `isFileLocked(path)`
- `scheduleDeleteOnReboot(path)`

### 6.8 ANSI、终端编码与控制台能力

建议归入 `platform/diagnostics/`：

- ANSI 颜色支持检测
- 终端编码探测
- Windows `cmd` / PowerShell 输出兼容性检查

建议预留接口：

- `isAnsiSupported()`
- `getTerminalEncoding()`

### 6.9 导出约定与路径别名

建议在阶段一就定下来：

- `platform/index.ts`
  - 只导出当前平台适配器选择逻辑与公共类型。
- 各子目录是否需要 `index.ts`
  - 建议需要，但只做薄封装，避免深层相对路径导入。
- `tsconfig.json`
  - 增加路径别名：
    - `@app/*`
    - `@core/*`
    - `@runtime/*`
    - `@platform/*`
    - `@packaging/*`
    - `@shared/*`

同时建议增加约束：

- `core/` 中禁止直接使用 `os.platform()`
- `core/` 中禁止直接拼平台目录
- `packaging/` 不允许反向依赖 `core/web-ui/client`

## 9. 分阶段实施计划

### 阶段一：建立平台抽象骨架

目标：不改变行为，只建立目录和依赖方向。

任务：

- 新增 `platform/types.ts`
- 新增 `platform/index.ts`
- 新增 `platform/windows.ts`
- 新增 `platform/macos.ts`
- 新增 `platform/linux.ts`
- 新增测试目录骨架 `tests/unit`、`tests/platform`、`tests/fixtures`
- 定义平台适配器 mock 约定

## 11. 风险

### 风险一：抽象过度

如果一开始接口设计过细，会让简单逻辑变复杂。  
应对方式：

- 第一版只抽最明显的平台差异
- 不一次性拆到很细的子模块

### 风险二：迁移过程中 import 大量变化

应对方式：

- 每阶段只处理一个主题
- 每次迁移后立即 `tsc` 校验

### 风险三：行为意外变化

应对方式：

- 第一阶段以“零行为变化”为硬目标
- 先加测试，再改行为

### 风险四：离线包诉求反向污染核心代码

应对方式：

- 离线包逻辑必须进入 `packaging/offline/`
- 不允许在 `core/` 里塞 release 打包细节

### 风险五：迁移期间兼容性问题与回滚困难

应对方式：

- 保留双轨过渡期
  - 新适配器入口可先包住旧实现
  - 核心服务先不删除旧内部函数
- 每一阶段都能单独回滚
- 在 release 前保留一段双轨验证期，而不是一次性切换

### 风险六：Windows 运行时差异被低估

应对方式：

- 把管理员权限、ExecutionPolicy、长路径、文件锁、信号处理明确列为 Windows 专项主题
- 不把这类问题当作普通“路径兼容性”附属项处理
- 让 `platform/windows.ts` 逐步承载完整 Windows 运行时适配职责

## 12. 验收标准

### 结构层面

- 业务核心文件不再直接散落平台判断。
- 平台相关逻辑集中在 `platform/`。
- 发布与离线包逻辑集中在 `packaging/`。

### 代码层面

- `index.ts` 只做装配，不再承载复杂平台逻辑。
- `deploy-task-service.ts` 不再直接负责平台工具选择。
- `self-update-service.ts` 不再直接持有平台路径细节。

### 交付层面

- Windows 专属 bug 修复时，修改范围主要落在 `platform/windows.ts` 及相关平台模块。
- 可在不污染核心逻辑的前提下新增 Windows 离线包。
- 日志、缓存、配置、密钥、临时目录都能由平台层统一解析。
- 测试可通过 mock 平台适配器隔离系统依赖。
- Windows 环境阻断项可在部署前明确诊断：
  - 管理员权限
  - ExecutionPolicy
  - 长路径支持
  - 文件锁
  - 子进程清理
  - ANSI/终端编码

### 工具链层面

- `tsconfig.json` 路径映射与目录结构一致。
- 具备 lint 约束，能阻止 `core/` 重新长出平台分支。
- 文档中目录、脚本、发布流程说明与代码实现一致。

## 13. 建议的第一批落地任务

如果按当前仓库节奏推进，第一批建议只做下面这些：

1. 新增 `platform/` 基础骨架
2. 拆出 `browser-utils.ts`
3. 拆出 `openclaw-project.ts` 中的路径逻辑
4. 拆出 `deployment-service.ts` 中的依赖安装策略
5. 拆出 `self-update-service.ts` 中的平台目标路径

这一批完成后，平台边界就会明显清晰，后续再做 release / offline bundle 才不会继续把共享逻辑弄乱。

## 14. 结论

这个项目当前不是没有平台结构，而是采用了“共享主干 + 到处散落的平台分支”。  
本计划的核心，不是把项目拆成三套，而是把这些散落的平台分支收拢到统一的平台层和打包层中。

只要按上述顺序推进，既能保持当前交付速度，也能逐步把 Windows、macOS、Linux 的维护成本降下来。
