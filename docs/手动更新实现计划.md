# 龙虾助手手动更新改造实施计划

## 1. 文档目的

这份文档不是策略说明，而是**可执行的实施计划**。  
目标是把龙虾助手从“启动前自动更新、部署前可能被更新阻塞”的模式，改成：

- 启动不阻塞
- 24 小时检查一次
- 普通版本手动更新
- 仅最低支持版本以下才强制更新

## 2. 现状判断

当前仓库里运行链路的关键点如下：

- 启动入口通过 [app/index.js](/Users/lylyyds/Desktop/openclaw-deploy/apps/cli/dist/app/index.js) 把 `ensureManagedSelfInstall` 和 `checkSelfUpdate` 注入 `bootstrapApp`
- 启动编排在 [bootstrap-service.js](/Users/lylyyds/Desktop/openclaw-deploy/apps/cli/dist/core/bootstrap/bootstrap-service.js)

- 本地配置存储在 [lobster-config.js](/Users/lylyyds/Desktop/openclaw-deploy/apps/cli/dist/lobster-config.js)
- Web UI 客户端逻辑分散在：

当前问题：

- `bootstrapApp()` 在 Web 控制台启动前就先调用 `ensureManagedSelfInstall()` 和 `checkSelfUpdate()`
- `checkSelfUpdate()` 会直接下载并尝试替换二进制
- 每 24 小时再次检查，但没有清晰的状态缓存模型
- 目前没有明确的 `minimumSupportedVersion` / `required` 模式

## 3. 改造目标

### 3.1 启动行为

- 启动时不再等待更新完成
- Web 控制台先打开
- 更新检查改成后台异步执行

### 3.2 更新行为

- 自动“检查”保留
- 自动“安装”取消
- 普通更新改为手动触发
- 仅 `required` 模式阻止高风险操作

### 3.3 平台行为

- macOS：只提供手动下载/打开下载页
- Windows/Linux：可保留“立即更新”能力，但不阻塞启动

## 4. 目标状态模型

建议新增并持久化：

```ts
type UpdateState = {
  currentVersion: string;
  latestVersion?: string;
  minimumSupportedVersion?: string;
  mode: 'up_to_date' | 'available' | 'recommended' | 'required';
  lastCheckedAt?: string;
  lastCheckSucceededAt?: string;
  lastCheckFailedAt?: string;
  lastError?: string;
  downloading?: boolean;
  updateReady?: boolean;
  downloadUrl?: string;
  notesUrl?: string;
  platformAction?: 'download' | 'self_update';
};
```

## 5. 本地持久化方案

建议新增本地文件：

- `update-state.json`

建议位置：

- 与龙虾助手现有配置目录同级或同目录
- 由 [lobster-config.js](/Users/lylyyds/Desktop/openclaw-deploy/apps/cli/dist/lobster-config.js) 统一提供路径

建议新增函数：

- `clearUpdateStateError()`

## 6. 最终行为定义

### 6.1 启动时

1. 读取主配置
2. 读取缓存的 `update-state.json`
3. 启动 Web 服务
4. 打开浏览器
5. 启动后异步执行 `maybeCheckForUpdates()`

### 6.2 `maybeCheckForUpdates()`

规则：

- 若当前不是打包运行态，直接跳过
- 若距离 `lastCheckedAt` 小于 24 小时，跳过网络检查
- 若用户手动点击“检查更新”，则忽略 24 小时限制

### 6.3 版本模式判断

建议逻辑：

- `current >= latest` -> `up_to_date`
- `current < latest && current >= minimumSupportedVersion` -> `available` 或 `recommended`
- `current < minimumSupportedVersion` -> `required`

### 6.4 用户点击更新时

- `macOS`
  - 打开下载页
  - 或下载后提示手动替换
- `Windows/Linux`
  - 执行 `performSelfUpdate()`
  - 成功后提示重启或自动重启

### 6.5 部署 OpenClaw 时

- 不再因 `available` 或 `recommended` 被阻塞
- 仅 `required` 时阻止：
  - 部署 OpenClaw
  - 更新 OpenClaw
  - 卸载/安装敏感操作

## 7.1 [bootstrap-service.js](/Users/lylyyds/Desktop/openclaw-deploy/apps/cli/dist/core/bootstrap/bootstrap-service.js)

### 当前问题

- 在服务器启动前执行：
  - `ensureManagedSelfInstall(...)`
  - `checkSelfUpdate(...)`

### 需要改动

- 保留 `ensureManagedSelfInstall(...)` 的平台固定安装副本逻辑
  - 但只处理“固定安装位置切换”
  - 不承担网络更新职责
- 删除启动前同步调用 `checkSelfUpdate(...)`
- 新增：
  - `loadCachedUpdateState(...)`
  - `maybeCheckForUpdates(...)`

### 目标结构

```js

server.listen(port, () => {
  // 启动 Web
  void deps.maybeCheckForUpdates({ force: false });
});
```

## 11. 测试清单

### 启动链路

- 首次启动时，Web 控制台能在更新检查前打开
- 网络异常时，启动不阻塞
- 24 小时内重复启动不重复打检查请求

### 更新状态

- 无缓存时正常创建
- 有缓存时正常读取
- 检查失败时保留旧状态并记录错误

### UI

- 能显示当前版本
- 能显示最新版本
- 能显示最后检查时间
- 能区分 `available / recommended / required`

### 平台行为

- macOS 点击更新不走自动替换
- Windows/Linux 点击更新可以执行安装

### 强制门槛

- `available` 不阻止部署
- `recommended` 不阻止部署
- `required` 阻止部署/更新 OpenClaw

## 12. 回滚方案

如果改造后出现问题，回滚顺序如下：

1. 保留新 `UpdateState` 文件，不删
2. 回退 UI 调用
3. 回退 API 新接口
4. 最后再回退启动解耦逻辑

这样即使回滚，也不会破坏已有配置和缓存文件。

## 13. 交付标准

完成后应满足：

- 启动不再因更新逻辑被阻塞
- 部署 OpenClaw 不再因普通版本可更新而被打断
- 更新检查结果可缓存 24 小时
- 用户能明确知道：
  - 当前版本
  - 最新版本
  - 是否建议更新
  - 是否必须更新
- macOS 不再伪装成自动更新已可用
- Windows/Linux 仍保留手动触发的更新能力
